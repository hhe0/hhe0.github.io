【MongoDB权威指南】读书笔记(三)

发表于 2018-09-07   |   更新于 2019-05-07 | 分类于 MongoDB读书笔记系列 | | 阅读次数

这部分主要介绍索引。

索引

介绍了什么是索引，如何合理地创建和删除索引以及如何对创建索引的效率进行评估。

索引简介

索引可以根据给定的字段组织数据，让MongoDB能够非常快地找到目标文档。

检查索引创建进度

MongoDB限制每个集合上最多只能有64个索引。通常，在一个集合上，不应该拥有两个以上的索引。

复合索引简介

一个建立在多个字段上的索引。

  • 如果结果集的大小超过32MB，MongoDB就会报错，拒绝对如此多的数据进行排序；
  • 可以使用explain()来查看MongoDB对查找的默认行为；
  • 可以使用hint()来强制MongoDB使用某个特定的索引。

使用复合索引

a. 选择键的方向

为了优化查询和排序，应当建立合适方向的索引。

b. 覆盖索引

  • 覆盖索引包含用户请求的所有字段，而不去返回真实的文档；
  • 如果在一个含有数组的字段上做索引，这个索引永远也无法覆盖查询。

c. 隐式索引

如果有一个拥有N个键的索引，那么你同时“免费”得到了所有这N个键的前缀组成的索引。注意这些键的任意子集组成的索引未必可用。

$操作符如何使用索引

a. 低效率的操作符

  • 无法使用索引的操作符：$where和检查一个键是否存在的$exists；
  • 通常来说，取反的效率是比较低的。$ne查询可以使用索引，但并不是很有效；
  • $not有时能够使用索引，但是通常它并不知道要如何使用索引。例如，它可以对一些范围表达式或者正则表达式进行翻转，但是大多数情况都会退化为全表扫描；
  • $nin总是进行全表扫描；
    如果需要快速执行一个这些类型的查询，可以试着找到另一个能够使用索引的语句，将其添加到查询中，这样就可以在MongoDB进行无索引匹配时先将结果集的文档数量减到一个比较小的水平。

b. 范围

设计基于多个字段的索引时，应该将会用于精确匹配的字段放在索引的前面，将用于范围匹配的字段放在最后。

c. OR查询

  • MongoDB在一次查询中只能使用一个索引；
  • $or查询实际上是执行了两次查询，然后将结果集进行合并，一般比单词查询的效率低。因此，尽量使用$in而不是$or；
  • 如果不得不使用$or，记住，MongoDB需要检查每次查询的结果集并且从中移除重复的文档；
  • 使用$in查询无法控制返回文档的顺序，除非显示指定。

索引对象和数组

MongoDB允许深入文档内部，对嵌套字段和数组建立索引。嵌套对象和数组字段可以与复合索引中的顶级字段一起使用。大多数情况下，它们与“正常”索引字段的行为是一致的。

a. 索引嵌套文档

对嵌套文档本身建立索引和对嵌套文档的某个字段建立索引是不同的，应当区分。

b. 索引数组

  • 对数组建立索引实际上是对数组中的每一个元素建立索引；
  • 需要考虑对整个数组建立索引还是对数组中的特定字段来建立索引；
  • 一个索引中的数组字段最多只能有一个：为了避免在多键索引中索引条目中爆炸性增长（笛卡尔积）。

c. 多键索引

  • 对于某个索引的键，如果这个键在某个文档中是一个数组，那么这个索引就会被标记为多键索引；
  • 一个多键索引explain()后的”isMultikey”字段值为true；
  • 索引只要被标记为多键索引，就无法再变成非多键索引了，即使这个字段为数组的所有文档都从集合中删除；
  • 要将多键索引恢复为非多键索引，唯一的方法是删除再重建这个索引；
  • 多键索引可能会比非多键索引慢一些。

索引基数

  • 基数就是集合中某个字段拥有不同值的数量；
  • 一般来说，应该在基数比较高的键上建立索引，或者至少应该把基数较高的键放在复合索引的前面（低基数的键之前）。

explain()和hint()

使用explain来诊断查询结果，hint来强制MongoDB使用一个特定的索引。

explain()返回的各个字段

  • “cursor” : “BtreeCursor age_1_username_1”
    BtreeCursor表示本次查询使用了索引，age_1_username_1表示使用了复合索引{“age” : 1, “user_name” : 1}。对结果进行逆序排序，会有”reverse”字段，”multi”表示是否使用多键索引；
  • “isMultiKey” : false
    用于说明本次查询是否使用了多键索引；
  • “n” : 8332
    本次查询返回的文档数量；
  • “nscannedObjects” : 8332
    按照索引指针去磁盘上查找实际文档的次数；
  • “nscanned” : 8332
    如果有使用索引，那么这个数字就是查找过的索引条目数量；
  • “scanAndOrder” : false
    是否在内存中对结果集进行了排序；
  • “indexOnly” : false
    是否只使用索引就能完成此次查询；
  • “nYields” : 0
    为了让写入请求能够顺利执行，本次查询暂停的次数；
  • “millis” : 91
    数据库执行本次查询所耗费的毫秒数；
  • “indexBounds” : {…}
    描述了索引的使用情况，给出了索引的遍历范围。

查询优化器

  • 如果一个索引能够精确匹配一个查询，那么查询优化器就会使用这个索引；
  • 否则的话，MongoDB会从可能的索引子集中为每次查询计划选择一个，这些查询计划是并行执行的。最早返回100个结果的就是胜者，其他的查询计划都会被中止。

何时不应该使用索引

  • 使用索引需要进行两次查找：一次是查找索引条目，一次是根据索引指针来查找相应的文档；
  • 极端情况下，如果查找需要返回集合内的所有文档，使用索引进行查找的次数将会是全表扫描的两倍，效率会明显比全表低很多；
  • 一般来说，如果查询需要返回集合内的30%的文档（或者更多），那就应该对索引和全表扫描的速度进行比较。

影响索引效率的属性

索引通常适用的情况 全表扫描通常适用的情况
集合较大 集合较小
文档较大 文档较小
选择性查询 非选择性查询

强制全盘扫描

可以使用$natural指定文档按照磁盘上的顺序排列。对于一个活跃的集合来说没有意义，因为集合中文档的位置可能会发生移动、变化，对于只存在插入的集合有用。总之，慎用。

索引类型

常见的有唯一索引和稀疏索引。

唯一索引

  • 用来确保集合的每一个文档的指定键都有唯一值；
  • 特别注意：如果一个文档没有对应的键，索引会将其作为null存储。所以如果对某个键建立了唯一索引，但插入了多个缺少该索引键的文档，由于集合已经存在一个该索引键为null的文档而导致插入失败；
  • 所有的字段都必须小于1024字节，才能包含在索引中。如果一个文档的字段由于太大不能包含在索引里，MongoDB不会返回任何错误或警告。也就是说，超过8KB大小的键不会受到唯一索引的约束：可以插入多个同样的8KB长的字符串。
在已有集合上创建索引的去重

稀疏索引

  • 可以配合唯一索引使用：允许插入多个键值为null的文档。基本上来说，MongoDB中的稀疏索引只是不需要将每个文档都作为索引条目；
  • 稀疏索引不必唯一。

索引管理

  • 所有的数据库索引信息都存储在system.indexes集合中；
  • 默认情况下，新建索引时，MongoDB会尽可能快地创建索引，阻塞所有对数据库的读请求和写请求，一直到索引创建完成。可以使用background选项修改这一默认行为，但是执行的过程中依然会对应用程序的性能造成比较大的影响。

特殊的索引和集合

介绍MongoDB中一些特殊的集合和索引类型：用于类队列数据的固定集合；用于缓存的TTL索引；用于简单字符串搜索的全文本索引；用于二维平面和球体空间的地理空间索引；用于存储大文件的GridFs。

固定集合

  • 使用类似循环队列结构存储的集合；
  • 固定集合的大小固定，必须显示创建；
  • 固定集合被占满时，如果再插入新文档，固定集合会自动将最老的文档从集合中删除；
  • 可以用于记录日志。

自然排序

对固定集合可以进行的一种特殊的排序：排序返回的结果集中文档的顺序就是文档在磁盘上的顺序，其实就是文档新旧（创建时间）的顺序。如此一来，即使没有显示标记文档创建时间，也可以按照创建时间返回文档的顺序。

循环游标

  • 循环游标是一种特殊的游标，当循环游标的结果集被取光后，游标不会被关闭。当有新文档插入到集合中时，循环游标会继续取到结果；
  • 由于普通集合并不维护文档的插入顺序，所以循环游标只能用在固定集合上。

没有_id索引的集合

在2.2版本之前，固定集合默认是没有”_id”索引的。我这边使用的是4.0.0版本，经过试验发现，现在创建的动态集合也都是有”_id”的。

TTL索引

  • 为集合中的每一个文档设置一个超时时间，一个文档到达域设置的时间就会被删除；
  • 为了防止活跃的会话被删除，可以在会话上有活动发生时将”lastUpdated”字段更新为当前时间；
  • MongoDB每分钟对TTL索引进行一次清理，所以不应该依赖以秒为单位的时间保证索引的存活状态；
  • 在一个给定的集合上可以由多个TTL索引；
  • TTL索引不能是复合索引，但是可以像“普通”索引一样用来优化排序和查询。

全文本索引

  • 文本搜索很快的一种索引；
  • 一个集合上最多只能有一个全文本索引，但是全文本索引可以包含多个字段；
  • 全文索引与“普通”的多键索引不同，全文本索引中的字段顺序不重要：每个字段都被同等对待，可以为每个字段指定不同的权重来控制不同字段的相对重要性。

使用GridFS存储文件

GridFS是MongoDB的一种存储机制，用来存储大型二进制文件。

GridFS的优点

  • 使用GridFS能够简化你的栈，可以用来代替独立的文件存储工具；
  • GridFS会自动平衡已有的复制或者为MongoDB设置的自动分片，所以对文件存储做故障转移或者横向扩展会更容易；
  • 当用于存储用户上传的文件时，GridFS可以比较从容地解决其他一些文件系统可能会遇到的问题；
  • 在GridFS中，文件存储的集中度会比较高。

GridFS的缺点

  • GridFS的性能比较低；
  • 如果要修改GridFS上的文档，只能先将已有文档删除，然后再将整个文档重新保存。
