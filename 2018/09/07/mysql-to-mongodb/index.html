Mysql到MongoDB的数据迁移

发表于 2018-09-07   |   更新于 2019-05-07 | 分类于 MongoDB随笔系列 | | 阅读次数

最近在做将Mysql的数据迁移到MongoDB的工作，有一些经验可以分享一下。作为MongoDB的初学者，可能会有错误或者不到位的地方，还望各位不吝指教。

写在数据迁移前

分析问题

在进行数据迁移前，我们需要考虑一些问题

  • Mysql在使用过程中遇到了哪些问题？MongoDB是否可以有效地解决这些问题？
  • 是否了解Mysql与MongoDB的区别？
  • 业务的读写频率？
  • 是否有足够的时间和精力来学习一项新的技术？

MongoDB的优点

  • 易于使用。使用『文档』来取代传统关系型数据库『行』，可以在一个文档中嵌入数组和子文档用于表达复杂的层次关系；针对键值对没有预定义模式，易于增加或者删除字段；
  • 易于扩展。使用『文档』的数据类型使它能够很容易地在多台服务器之间进行数据分割，实现横向扩展；
  • 功能丰富。提供包括索引、聚合、特殊的集合类型（用于记录日志文件）、文件存储等独特功能；
  • 性能卓越。只要有可能，数据库服务器就会将处理和逻辑交给客户端（通过驱动程序或者用户的应用程序代码来实现）。

不适合使用MongoDB的场景

  • MongoDB不支持事务，对事务性有要求的应用程序不建议使用MongoDB；
  • 在多个不同维度上对不同类型的数据进行连接，这是关系型数据库擅长的事情，不建议使用MongoDB。

简单介绍一下我进行数据迁移的项目背景：当时表里面的数据量在以后可能会非常庞大；数据表需要满足较强的可扩充性；需要使用地理空间索引。这些原因促使我们需要将部分业务转移到MongoDB中，由此便开始了学习MongoDB的征程~

写在数据迁移时

如果你看完了上面这些还是坚持要迁移到MongoDB中，那么可能从现在开始你要转换看待数据的方式了。因为NoSQL与SQL的如此之大，不仅在于它们的增、删、改、查等基本操作，还有设计哲学根本的不同。

分析问题

同样地，我们需要在迁移时需要考虑这些问题

  • 我们如何在MongoDB中组织以前存储在Mysql的数据？一个集合对应一个数据表？多个集合对应一个数据表？一个集合对应多个数据表？
  • 数据类型的问题，比如我遇到的日期类型的数据；
  • 如何兼容线上用户进行存取数据？

范式化与反范式化

  • 范式化：将数据分散到多个不同的集合，不同集合之间可以相互引用数据；
  • 反范式化：将每个文档所需的数据都嵌入在文档内部。每个文档都拥有自己的数据副本，而不是所有文档共同引用同一个数据副本；
  • 决定何时使用范式化何时采用反范式化是比较困难的。范式化能够提高数据写入速度，反范式化能够提高数据读取速度。需要根据自己应用程序的实际需要仔细权衡。

内嵌数据和引用数据

更适合内嵌 更适合引用
子文档较小 子文档较大
数据不会定期改变 数据经常改变
最终数据一致即可 中间阶段的数据必须一致
文档数据小幅增加 文档数据大幅增加
数据通常需要执行二次查询才能获得 数据通常不包含在结果中
快速读取 快速写入

其实选择合理的存储方式是件很头疼的事情，很多时候还是要根据业务具体情况具体对待。

datetime类型的数据取出来也就是当字符串处理了，那么我们是不是要将这样的字符串直接存入MongoDB中？思考过后存在以下几种方案。

使用Date()类型存储

官方推荐的日期存储方式。优点是方便各种处理；缺点是各种数据库工具展示的时候都差8个小时（这与时区有关系），而且数据返回到前端时需要进行额外的处理。

使用字符串存储

MongoDB支持字符串使用字典序查询，但是这样的方式依赖日期存储的格式，比如存储为2018-09-04 10:00:00可以，但是存储为04/09/2018 10:00:00查询则会出现问题，而且查询效率也有问题。好处是数据直观。

使用时间戳存储

使用时间戳的好处是便于存储，但是在数据库中查看不够直观，可以考虑再加一个字段来存储时间戳的字符串表示。不过要按照时间戳做数据统计的话，哈哈哈哈哈。

兼容存取逻辑

因为业务不要求实时，我是直接停止数据插入，然后进行数据迁移的。那如果是实时的应用，你可能需要在一段时间内同时插入Mysql和MongoDB，取数据时优先从MongoDB，取不到则从Mysql这种方式。

写在数据迁移后

数据迁移完成后，总是需要对解决问题的过程做一番梳理。不免感叹，我在大学学习SQL时候竟然没有对关系型数据库提出过质疑：使用关系型数据库会不会遇到一些问题？是不是还有其他组织数据的方式？学习、使用MongoDB的过程有点像带着镣铐跳舞的我一下子脱下镣铐竟然。。跳起舞来有点别扭。。。道阻且长，后面有些学习心得会陆续分享，欢迎持续关注。
