通过一个电商数据模型和实现必要的CRUD操作来说明MongoDB的文档数据模型、查询语言和CRUD操作。

面向文档的数据

介绍面向文档的数据模型以及如何在MongoDB数据库、集合和文档级别里组织这些数据。

schema设计原则

  • 数据库schema设计是基于数据库特性、数据属性和应用系统选择最好的数据表示形式的过程；
  • 对于关系型数据库，在设计的时候遵循数据库设计范式即可，它通常可以确保通用查询以及数据一致性。

MongoDB在进行数据库系统建模时，需要考虑的问题

  • 应用访问的模式是什么？具体来说，读/写的比率是什么？查询是不是很简单？查询一个key还是更复杂的key？是否需要聚合查询？数据量是多少？
  • 数据的基本单位是什么？
  • 数据库的功能是什么？MongoDB不支持join连接查询，不支持事务；
  • 如何记录生成好的唯一ID或者主键？

设计电商网站数据模型

  • 一些ORM框架无法与RDBMS关系型数据库高效地结合，但是无法与MongoDB一起使用。这是因为文档本身已经是类对象表示形式了。其次是MongoDB驱动的原因，它已经提供了对MongoDB的高级别抽象了。毫无疑问，我们只使用驱动接口来构建MongoDB应用。

核心概念：数据库、集合、文档

数据库

数据库是集合和索引的命名空间和物理分组。

管理数据库
  • 第一次指定数据库时，不会显示创建数据库，只有往集合中插入数据的时候才会创建；
  • 删除数据库的时候要格外小心，因为没有办法回滚操作，它会从磁盘上删除数据库文件。
数据库文件和分配
  • 当创建数据库的时候，MongoDB会在磁盘上分配一系列数据库文件集合，包括所有的集合、索引，还有其他元数据；
  • 鉴于元数据的大小，这意味着数据库中索引和集合的数目的总和不能超过26000；
  • 随着持续添加数据到数据库，MongoDB会持续分配更多的数据文件。每个新数据文件会是前一个数据文件的两倍大小，直到最大的文件达到2GB的上限为止。

集合

集合是结构或概念上相似的文档的容器。

管理集合
盖子集合（固定集合）

盖子集合不允许正常集合的所有操作。例如，不允许从盖子集合里删除单个文档，而且也不能增加文档的大小。盖子集合最初是为日志设计的，所以没有必要实现删除和更新文档操作。

TTL集合
  • 允许为集合中的文档设置生存时间，超过生存时间后会自动删除；
  • TTL索引只会比较索引值和当前值的差别，因此甚至可以把索引值设置为将来的时间点。
系统集合
  • MongoDB存在2个系统集合，system.namespaces和system.indexes；
  • system.namespaces可以查询当前数据库定义的所有命名空间；
  • system.indexes存储了当前数据库里的定义的所有索引。

文档和插入

文档序列化、类型和限制
  • 所有的文档在发送给MongoDB之前都序列化为BSON格式，以后再从BSON反序列化；
  • 文档的key值也会存储在文档里而影响文档的大小。
字符串
  • 所有的字符串必须使用UTF-8编码。
数字
  • BSON指定了三种数据类型：double、int、long。
时间
  • BSON的实际类型用来存储临时值。从Unix纪元计时开始时间值使用64b整数的毫秒值表示，负数表示之前的时间。
文档的限制
  • BSON文档大小限制为16MB。一方面是因为性能问题，另一方面是组织开发者创建无意义的数据模型；
  • MongoDB文档的嵌套深度最大值限制是100。
大量插入（批量插入）
  • 大量插入对于性能非常有帮助。这意味着只需要一个连接就可以插入大量数据，而不需要大量独立地来回往返；
  • 大量插入具有一个限制：如果要插入100万文档，我们就必须把它们分隔为多个大量插入的文档组。

MongoDB的查询语言

查询条件和选择器

  • 查询条件允许我们使用单个或者多个查询选择器来指定查询结果；
  • 在MongoDB中所有的文本字符串匹配都是区分大小写的。如果需要不区分大小写匹配，可使用正则表达式。

选择器匹配

范围

  • 范围查询匹配值仅仅当它们含有相同类型的值时才进行比较，注意区分MongoDB对于不同类型的键值进行排序的处理方式。

设置运算符

  • 使用$in的另一种方法是作为布尔的包容性或者对单一属性；
  • 记住$in和$all可以使用索引，但$nin不能因此而要集合浏览；
  • 如果我们使用$nin，就尝试用它结合另一种使用该索引的查询项。

集合操作符总结

操作符 描述
$in 如果任意参数在引用集合里，则匹配
$all 如果所有参数在引用集合里且被使用在包含在数组的文档中，则匹配
$nin 如果没有参数在引用的集合里，则匹配

布尔运算符

  • $ne并不等同于运算符，最佳的使用方式是结合一个其他的运算符；否则，会因为不是使用索引而造成效率低下；
  • $ne适用于关键字指向的单一值或者数组；
  • $ne匹配指定值，$not不匹配MongoDB的运算符或者正则表达式的查询结果；
  • $or表示使用两个不同的关键字逻辑分离两个值。如果可能值的范围是相同的关键字，就是用$in替代；
  • 我们仅当不能使用更简单的方式来表示AND关系时才使用
  • $and

布尔运算符总结

操作符 描述
$ne 不匹配参数条件
$not 不匹配结果
$or 有一个条件匹配就成立
$nor 所有条件都不匹配
$and 所有条件都匹配
$exists 判断元素是否存在

查询关键字文档

  • 这个运算符是必须的，因为集合不会强制执行一个固定的模式，所以我们偶尔需要一种方式来查询包含特定关键字的文档。

匹配子文档

数组

  • 从逻辑上讲，仅当我们需要匹配多个或者更多属性的子文档才使用$elemMatch。

数组操作符

操作符 描述
$elemMatch 如果提供的所有词语在相同的子文档中，则匹配
$size 如果子文档数组大小与提供的文本值相同，则匹配

JavaScript查询

  • JavaScript表达式不能使用索引，并且带来大量消耗，因为它们必须在JavaScript解释器的上下文中评估并且是单线程的；
  • 当使用其他查询运算符无法表示查询时，就应该使用JavaScript查询；
  • 使用JavaScript表达式时至少结合一个其他查询运算符以削减结果集，减少必须加载到JavaScript上下文的文档数目。

正则表达式

  • MongoDB使用Perl兼容正则表达式编译，对正则表达式提供了全面的支持。

其他查询运算符

其他运算符总结

运算符 描述
$mod[(quotient), (result)] 如果元素除以除数结果则匹配
$type 如果元素的类型符合指定的BSON类型则匹配
$text 允许在建立文本索引的字段上执行文本搜索

文档更新概要

更新文档有两种方式：完整替换当前的文档，或者使用修改器修改文档的某个字段。

两种方式的比较

  • 替换是更通用的做法；
  • 目标更新通常可以获得更好的性能；
  • 目标操作允许原子更新文档。

原子文档处理

  • findAndModify命令允许我们在同一个往返过程中原子更新文档并返回它。原子更新就是一个不会被其他更新中断或者与其他操作交互的操作。
