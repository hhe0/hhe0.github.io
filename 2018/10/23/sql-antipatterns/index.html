SQL反模式(一)：乱穿马路

发表于 2018-10-23   |   更新于 2019-05-07 | 分类于 SQL读书笔记系列 | | 阅读次数
字数统计: | 阅读时长 ≈

程序员通常使用逗号分隔的列表来避免在多对多的关系中创建交叉表，我将这种设计方式定义为一种反模式，称为『乱穿马路』，因为乱穿马路也是避免过十字路口的一种方式。每个值都应该存储在各自的行与列中。

目标：存储多值属性

  • account_id原先的设计是存储单个值，构成一对一的关系。现在需要需要扩充为一对多的关系。

反模式：格式化的逗号分隔列表

  • 扩充方法：将account_id类型修改为varchar，并且将每个账号ID通过逗号进行分隔。

查询指定账号的产品

  • 查询时无法使用等号，而需使用正则匹配。匹配的表达式可能返回错误的结果，并且无法享受索引带来的性能优势；
  • 由于正则匹配的语法在不同的数据库中存在区别，因此SQL代码移植性不好。

查询指定产品的账号

  • 查询极不优雅；
  • 无法使用索引，效率很差。

执行聚合查询

  • 表达不清晰且复杂；
  • 有些表达式无法准确表达。

更新指定产品的账号

  • 无法指定更新的顺序；
  • 如果希望删除一个条目，则需要执行两次SQL语句：先提取老的列表，然后存储更新后的列表。

验证产品ID

  • 无法防止用户输入非法字段。

选择合适的分隔符

  • 如果字段本身包含分隔符，则可能出现潜在的问题。

列表长度限制

  • 由于varchar长度有限，且每个字段的长度大小长度不同，无法准备知道该字段可以存储多少个账户ID。

如何识别反模式

  • 列表最多支持存放多少数据？
  • 你知道在SQL中如何做分词查找吗？
  • 哪些字符不会出现在任何一个列表条目中？

合理使用反模式

  • 出于性能优化的考虑，可能在数据库的结构中需要使用反规范化的设计。将列表存储为以逗号分隔的字符串就是反规范化的一个实例；
  • 应用程序可能会需要逗号分隔的这种存储格式，也可能没必要获取列表中的单独项；
  • 谨慎使用反规范化的数据库设计。

解决方案：创建一张交叉表

  • 将account_id存储在一张单独的表中，而不是存储在products表中，从而每个独立的account值都可以占据一行。

通过账号查询产品和反过来查询

  • 使用索引提高了联结查询的效率。

执行聚合查询

更新指定产品的相关联系人

验证产品ID

  • 可以使用外键来验证数据；
  • 也可以使用SQL中的数据类型来约束条目。

选择分隔符

  • 无需分隔符。

列表长度限制

  • 如果需要添加限制，可以在应用程序中添加限制。

其他使用交叉表的好处

  • 查询效率高；
  • 方便添加其他属性。
