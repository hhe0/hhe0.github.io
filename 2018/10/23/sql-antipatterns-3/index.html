SQL反模式(三)：需要ID

发表于 2018-10-23   |   更新于 2019-05-07 | 分类于 SQL读书笔记系列 | | 阅读次数
字数统计: | 阅读时长 ≈

规范仅仅在它有帮助时才是好的，这里指的是ID。

目标：建立主键规范

  • 主键是数据库确保数据行在整张表中唯一性的保障，它是定位到一条记录并且确保不会重复存储的逻辑机制。主键也同时可以被外键引用来建立表与表之间的关系；
  • 大多数表中的每个属性的值都有可能被很多行使用。在这样的表中，需要引入一个而对于表的域模型无意义的新列来存储一个伪值。这一列被用作这张表的主键，从而通过它来确定表中的一条记录，即便其他的列允许出现适当的重复项。这种类型的主键列我们通常称为伪主键或者代理键。

反模式：以不变应万变

  • 认为主键等于id。

冗余键值

  • 可能存在其他的列id相比更合适作为主键。

允许重复项

  • 当我们把bug_id定义为主键时，此时的(bug_id, product_id)便会允许重复；
  • 当我们为(bug_id, product_id)建立unique约束时，此时的id便又失去了意义，成了多余列。

意义不明的关键字

  • 我们使用主键来唯一的定位一条记录，因此主键的列名就应该更加便于理解。

使用USING关键字

  • 如果所有的表都要求定义一个叫做id的伪主键，那么作为外键的列将永远不能使用和引用的列相同的列名。同时，每次查询都必须使用啰嗦的ON表达式。

使用组合键之难

  • 如果要比较两个键值，必须比较其所包含的所有列的值；
  • 一个引用组合键的外键，其本身也必须是一个组合外键。

如何识别反模式

  • 我觉得这张表不需要主键；
  • 我怎么能在多对多的表中存储重复的项？
  • 我学过数据库设计理论，里面说我应该把数据移到一张查询表中，然后通过ID查找。但是我不想这么做，因为每次我都想获得真实的数据，都不得不做一次联结查询。

合理使用反模式

  • 使用伪主键，或者通过自动增长的整型的机制本身没有什么错误，但不是每张表都需要一个伪主键，更没有必要将每个伪主键都定义为id；
  • 对于太长而不方便实现的自然键来说，伪主键是很好的替代品。

解决方案：裁剪设计

  • 主键是约束而非数据类型。你可以等定义任意列或任意多的列为主键，只要其数据类型支持索引。同时，还可以将一个列的数据类型定义为自增长的整型而不设定其为主键。这两者是完全无关的。

直截了当地描述设计

  • 为主键选择更有意义的名称；
  • 外键应该尽可能地和所引用的列使用相同的名称；
  • 有时外键的名称需要和其所引用的主键区分开，从而使得它们之间的引用关系表现得更加清晰。

打破传统

  • 为每一列指定一个有意义的名称。

拥抱自然键和组合键

  • 如果你的表中包含一列能确保唯一、非空以及能够用来定位一条记录，就别仅仅因为传统而觉得有必要再加上一个伪主键；
  • 在核实的情况下也可以使用组合键；
  • 值得注意的是，一个引用了组合键的外键同样需要时一个组合键。这看上去像是很累赘地对其依赖的列做了一个副本，但这样做也有好处，它能简化原来需要一个lian
