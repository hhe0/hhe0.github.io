SQL反模式(五)：实体-属性-值

发表于 2018-10-23   |   更新于 2019-05-07 | 分类于 SQL读书笔记系列 | | 阅读次数
字数统计: | 阅读时长 ≈

为元数据使用元数据。

目标：支持可变的属性

  • 通常来说，一张表由一些属性列组成，表中的每一条记录都使用这些列，因为每条记录表示的都是相似的对象实例。不同的属性集合表示不同的对象，因而就应该用不同的表来区分。

反模式：使用泛型属性表

  • 实体：通常来说这就是一个执行父表的外键，父表的每条记录表示一个实体对象；
  • 属性：在传统的表中，属性即每一列的名字，但在这个新的设计中，我们需要根据不同的记录来解析其标识的对象属性；
  • 值：对于每个实体的每一个不同属性，都有一个对应的值；
  • 这种设计称为实体——属性——值，简称EAV。有时也称之为：开放架构、无模式或者名-值对。

EAV设计的好处

  • 表的列数都很少；
  • 新增的属性不会对现有的表结构造成影响，不需要新增列；
  • 避免了由于空值而造成的表内容混乱。

查询属性

支持数据完整性

  • EAV的设计方式，需要放弃很多传统的数据库设计所带来的方便之处。

无法声明强制属性

  • 无法使用SQL的类型约束，只能通过外部程序的代码来确保属性值是否一定存在。

无法使用SQL的数据类型

  • 为了兼容各种数据类型，无法使用SQL的数据类型进行约束。

无法确保引用完整性

  • 由于存储的数据可能来自于各个表导致的问题。

无法配置属性名

  • 可以通过将attr_name声明为一个外键，指向一张存储着所有可能出现的属性名的表。然而，这不支持在运行时定义的各种属性，即使那是EVA设计的一个非常普遍的使用方式。

重组列

  • 随着属性的数量不断增多，联结的数量也不增长，查询的开销也成指数级地增长。

如何识别反模式

  • 数据库不需要修改元数据就可以扩展。你还可以在运行时定义新的属性。
  • 查询时我能用的最大数量的联结是多少？
  • 我想象不出怎么为我们的电子商务平台生成报告。我们需要雇一个顾问来做这件事。

合理使用反模式

  • 大多数应用程序仅仅在有限的几张表甚至于仅一张中需要存储无范式的数据，而其他的数据需求适用于标准的表的设计。如果你明白在你的项目中使用EAV设计的风险和你要做的额外工作，并且谨慎地使用它，它的副作用会变得很小；
  • 哪些富有经验的数据库顾问给出的报告显示，使用EAV设计的系统在一年以内就会变得极其笨重。

解决方案：模型化子类型

  • 具体案例具体分析。

单表继承

  • 将所有相关的类型都存在一张表中，为所有类型的所有属性都保留一列。同时，使用一个属性来定义每一行表示的子类型。对于所有的子类型来说，既有一些公共属性，但同时又有一些子类型特有属性。这些子类型特有属性必须支持空值；
  • 当程序需要加入新对象时，很可能需要在原来表中增加新的属性列来适应这些新对象；
  • 当数据的子类型很少，以及子类型特殊属性很少，并且你需要使用Active Record模式来访问单表数据库时，单表继承模式是最佳选择。

实体表继承

  • 为每个子类型创建一张独立的表。每个表包含那些属于基类的共有属性，同时也包含子类型特殊化的属性；
  • 实体继承设计相比单表继承设计的优势在于提供了一种方式，让你能阻止在一行内存储一些和当前子类型无关的属性，而且也不用像在单表继承设计里那样使用额外的属性来标记子类型；
  • 实体继承的缺点是如果将一个新的属性增加到通用属性中，必须为每个子表都加一遍；
  • 当你很少需要一次性查询所有子类型时，实体继承表设计是最好的选择。

类表继承

  • 类表继承模拟了面向对象中的继承，把表当成面向对象中的类。创建一张基类表，包含所有子类型的公共属性。对于每个子类型，创建一个独立的类，通过外键和基类表相连；
  • 基类表和子类表之间一对一的关系由元数据来确保，因为子类型表的外键也同样是主键，因而就必须是唯一的；
  • 当你经常要查询所有子类型时这个设计是最佳选择，引用这些公共列就行了。

半结构化数据模型

  • 如果你有很多子类型或者你必须经常地增加新的属性支持，那么可以使用一个BLOB列来存储数据，用XML或者JSON格式——同时包含了属性的名字和值；
  • 这个设计的优势之处在于其优异的扩展性，可以添加尽可能多的子类型；
  • 该设计的缺点在于SQL基本上没有办法获取某个指定的属性。无法在一行BLOG字段中简单地选择一个独立的属性，并对其进行限制、聚合运算、排序等其他操作。你必须获取整个字段机构并通过程序去解码并且解释这些属性。（事实上，目前Mysql已经增加了对于JSON`格式的支持，但是基于字段的操作依旧鸡肋，效率堪忧）。

后处理

  • 别尝试像在传统表中那样写查询语句，将实体当成单行数据读取。取而代之的是，查询和实体关联的属性并且将这些行组合在一起，就像它们存储的结构一样。
