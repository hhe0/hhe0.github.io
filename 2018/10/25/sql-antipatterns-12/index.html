SQL反模式(十二)：乱用索引

发表于 2018-10-25   |   更新于 2019-05-07 | 分类于 SQL读书笔记系列 | | 阅读次数
字数统计: | 阅读时长 ≈

了解你的数据，了解你的查询需求，然后MENTOR你的索引。

目标：优化性能

  • 改善性能最好的技术就是在数据库中合理地使用索引。

反模式：无规划地使用索引

  • 不使用索引或索引不足；
  • 使用了太多的索引或者使用了一些无效索引；
  • 执行了一些让索引无能为力的查询。

无索引

  • 索引能够通过带给你更多的好处来抵消它的额外开销；
  • 索引可以帮助UPDATE或者DELETE语句快速地找到对应的记录；
  • 一条在非索引列上执行的语句会导致数据库执行全表遍历来查找匹配记录。

索引过多

  • 无需为主键显式建立索引；
  • 对于长字符串，这种类型的索引要比更为紧凑数据类型的索引大很多；
  • 组合索引中的列的顺序很重要：你需要在查询条件、联合条件或者排序规则上使用定义索引时的从左到右的顺序。

索引也无能为力

  • 如果某次基于索引的查询结果是分散在整张索引表的，则该次查询则很有可能无法使用索引。

如何识别反模式

  • 这是我的查询语句，我要怎样才能让它更快？
  • 我在每个字段上都定义了索引，为什么它没有变得更快？
  • 我听说索引会使数据库变慢，所以我不会使用它。

合理使用反模式

  • 分离率是衡量数据库索引的一个指标。它是一张表，所有不重复的值的数量和总记录条数之比。分离率越低，索引的效率也就越低；
  • 如果一个给定的值出现在这张表的很多条记录中，查询索引比简单地扫描一遍一张表更麻烦。事实上，使用这个这个索引的开销可能比不使用它还大；
  • 尽量去尝试不同的索引。

解决方案：MENTOR你的索引

  • 测量(Measure)，解释(Explain)，挑选(Nominate)，测试(Test)，优化(Optimize)和重建(Rebuild)；
  • 在对性能问题下结论之前，先用一些分析工具来测试一下是否是数据库的原因。否则你可能就在做一些过度优化。

测量

  • 大多数数据库都提供了一些方法来记录执行SQL查询的时耗，因此可以以此来定位最耗时的查询；
  • 在部署程序之后，你可通过进行profile分析来得到更准确的信息。

解释

  • 让数据库生成一份它所做分析的报表，称之为查询执行计划。

挑选

  • 在查询执行计划中查找没有使用索引的查询操作。

测试

  • 在创建完索引之后，需要重新跟踪那些查询。需要确认你的改动确实提升了性能，然后就能确定工作完成了。

优化

  • 数据库服务允许你配置缓存所需要的系统内存大小。

重建

  • 索引在平衡的时候其效率最高，当你更新或者删除记录时，索引就逐渐变得不平衡。就如同文件系统随着时间的推移会产生很多磁盘碎片一样；
  • 如果想要最大限度地使用索引，就需要定期对索引进行维护；
  • 最重要的规则是千万别瞎猜索引的使用方法。
