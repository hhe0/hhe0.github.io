【深入PHP：面向对象、模式与实践(五)】读书笔记

发表于 2018-10-18   |   更新于 2019-05-07 | 分类于 PHP读书笔记系列 | | 阅读次数

本章将介绍一些关键的面向对象的设计原则。

模式的启示

  • 继承和封装是一种定义和构建系统的新途径，而多态就是一个类型的实现在运行时进行了切换；
  • 组合优于继承。

组合与继承

  • 继承是应对变化的环境及上下文设计的有效方式，然而它会限制灵活性，尤其当类承担多重责任的时候。

问题

使用组合

  • 显示调用另一个对象的方法来执行一个请求的方式便是所谓的『委托』；
  • 组合使用对象比使用继承体系更灵活，因为组合可以以多种方式动态地处理任务，不过这可能导致代码的可读性下降。因为组合需要更多的对象类型，而这些类型的关系并不像在继承关系中那般有固定的可预见性，所以要理解系统中类和对象的关系会有些困难。

解耦

问题

  • 当我们看到系统中的一个组件迫使系统其它许多地方也发生改变的时候，就可判断为紧耦合了。

降低耦合

  • 将组件从系统中分离，达到降低耦合的目的。

针对接口编程，而不是针对实现编程

  • 可以把不同的实现隐藏在父类所定义的共同接口下。然后客户端代码需要一个父类的对象而不是一个子类的对象，从而使客户代码可以不用关心它实际得到的是哪个具体实现；
  • 如何使用参数类型提示来调整参数对象的“通用性”是需要仔细权衡的。选择过于通用，则会降低方法的安全性。而如果需要某个子类型的特有功能，那么方法接受另一个子类类型则可能会有风险。

变化的概念

  • 把变化的概念封装起来。

封装的好处

  • 专注于职责；
  • 通过组合提高灵活性；
  • 使继承层级体系更紧凑和集中；
  • 减少重复。

如何发现变化的元素

  • 包括一次实现不同分支的继承；
  • 子类化某个算法，而该算法对于该对象类型的核心职责是偶然的；
  • 出现了条件表达式。

父子关系

  • 你还不需要它——You aren’t going to need it.
  • 用最简单的方式来完成任务——Do the simplest thing that works.

模式

用于生成对象的模式

  • 这类模式关注对象的实例化。

用于组织对象和类的模式

  • 这类模式帮助我们组织对象的组成关系。

面向任务的模式

  • 这类模式描述了如何让类和对象合作来达成特定目标。

企业模式

  • 这类模式描述了典型因特网编程问题和解决问题的模式。

  • 这类模式描述数据库存取数据及对象——数据库映射的相关模式。
